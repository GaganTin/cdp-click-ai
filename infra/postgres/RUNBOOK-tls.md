# Enable & require TLS on `cdp-ai-postgres` (public LoadBalancer)

Goal: encrypt all traffic to the publicly-exposed Postgres (`20.24.249.154:5432`,
ns `cdp-ai`). Currently `ssl = off`, `password_encryption = scram-sha-256`,
LB locked to `loadBalancerSourceRanges` `74.220.50.0/24`, `74.220.58.0/24`.

Each `helm upgrade` here restarts the single primary pod (~30–60s connection blip).
Do it in a maintenance window. **Two phases** so clients are never locked out.

## Pre-flight
- Note the current revision for rollback: `helm history cdp-ai-postgres -n cdp-ai`
  (currently revision **2** = the working, no-TLS state).
- Rollback escape hatch at any point: `helm rollback cdp-ai-postgres 2 -n cdp-ai`.

## Phase 1 - turn TLS ON, do NOT enforce yet
Edit `values-tls.yaml`: in `pgHbaConfiguration`, use the **`host`** (not `hostssl`)
forms for the `0.0.0.0/0` and `::/0` lines (see the PHASE 1 comment in the file).
Then:

```sh
helm upgrade cdp-ai-postgres <same-chart-source> --version 18.6.7 \
  -n cdp-ai --reuse-values -f infra/postgres/values-tls.yaml
kubectl rollout status statefulset/cdp-ai-postgres -n cdp-ai
```

Verify SSL is now available (run from the pod):
```sh
kubectl exec -n cdp-ai cdp-ai-postgres-0 -- bash -c \
  'PGPASSWORD=$(cat $POSTGRES_POSTGRES_PASSWORD_FILE) psql -U postgres -h 127.0.0.1 -tAc "show ssl;"'
# expect: on
```
At this point old `sslmode=disable` clients still work - nothing is broken yet.

## Phase 2 - flip the clients to SSL
node-postgres and libpq read `sslmode` DIFFERENTLY against a self-signed cert:

| Client | What to set | Why |
|---|---|---|
| Node app / scripts (`POSTGRESQL_CONN` in `.env`) | `?sslmode=no-verify` | node-postgres treats `require` as verify-the-cert → fails on self-signed. `no-verify` = encrypt, skip verify. |
| Airflow Variable `cdp_ai_app_pg_conn` (psycopg2/libpq) | `?sslmode=require` | libpq `require` = encrypt, no cert verify. Works with self-signed. |

1. **App** - `.env`:
   ```
   POSTGRESQL_CONN=postgresql://cdpAdmin:<pw>@20.24.249.154:5432/app?sslmode=no-verify
   ```
   Restart the Node server (e.g. `pm2 restart <app>`).

2. **Airflow** - Admin → Variables → `cdp_ai_app_pg_conn`, append `?sslmode=require`
   to the DSN (or `&sslmode=require` if it already has query params). Affects all
   GA/GSC/commerce/attributes DAGs (they all use `lib/config.get_app_pg_conn_kwargs`).

Confirm every live connection is now encrypted:
```sh
kubectl exec -n cdp-ai cdp-ai-postgres-0 -- bash -c \
  'PGPASSWORD=$(cat $POSTGRES_POSTGRES_PASSWORD_FILE) psql -U postgres -h 127.0.0.1 -tAc \
   "select a.usename, a.client_addr, s.ssl from pg_stat_ssl s join pg_stat_activity a on s.pid=a.pid where a.client_addr is not null;"'
# every network client should show ssl = t
```

## Phase 3 - ENFORCE (reject non-SSL)
Only after Phase 2 shows all network clients on `ssl = t`. Edit `values-tls.yaml`
back to the `hostssl` forms (the default in the committed file), then re-run the
same `helm upgrade` + `rollout status`. Non-SSL connections are now refused.

Verify enforcement - a plaintext attempt must fail:
```sh
# from a machine inside an allowed source range:
psql "host=20.24.249.154 port=5432 dbname=app user=cdpAdmin sslmode=disable"   # expect: rejected
psql "host=20.24.249.154 port=5432 dbname=app user=cdpAdmin sslmode=require"   # expect: connects
```

## Follow-ups (not in this change)
- `verify-full` needs a cert whose SAN includes `20.24.249.154` (or a DNS name) and
  a stable CA - autoGenerated won't satisfy it. Use cert-manager or a provided secret.
- Consider rotating the `cdpAdmin` password (it's in `.env` + helm values in plaintext)
  and giving the app + Airflow their own least-privilege roles instead of `cdpAdmin`.
